# ODF Authoring — Canonical Path

Reference for producing `.odt` (text), `.ods` (spreadsheet), and `.odp` (presentation) deliverables under the Spacecraft Software document-format skill. Load this **alongside** `markdown-companion.md` for any ODF task.

The Spacecraft Software Standard §9/§10 palette and typography rules are summarised in SKILL.md §4 / §5 / §7 — this file focuses on how to **apply** them in ODF.

## §A — Why ODF first

- **Open ISO standard** (ISO/IEC 26300). No vendor lock-in, fully documented XML format.
- **Native to LibreOffice** — every Spacecraft Software tool ships compatible with LibreOffice; round-trips are loss-free.
- **Accepted by Google Workspace** — Drive accepts ODF uploads and the converter is mature for `.odt`. Sheets and Slides also accept `.ods` / `.odp` with reasonable fidelity.
- **Opened by Microsoft Office 2007 SP2 and later** with shape-specific fidelity drift (see §B). MS Office 365 and Office 2021+ open ODF cleanly enough for most internal/external sharing.
- **Diff-friendlier than MS Office** in version control — unzipped `content.xml` is plain XML with stable element ordering when generated by LibreOffice or `odfpy`.

## §B — Cross-suite compatibility matrix

Approximate fidelity expectations when an ODF file is opened in a non-LibreOffice reader. "Fidelity" here means visual + structural preservation; raw text always survives.

| Shape  | LibreOffice | Google Docs / Sheets / Slides | MS Office (Word / Excel / PowerPoint) |
|--------|-------------|-------------------------------|----------------------------------------|
| `.odt` | 100%        | ~95%; font fallback for unhosted faces; embedded fonts ignored at view but preserved on download | ~90%; page background sometimes rendered as watermark frame in Word ≤ 2016 |
| `.ods` | 100%        | ~85%; basic formulas survive, conditional formatting partial, chart styling resets | ~85%; `OFFSET`/`INDIRECT` round-trip cleanly, `IFERROR` semantics on legacy Excel can differ, named ranges preserved |
| `.odp` | 100%        | ~80%; slide background colour survives, slide-master themes partially flatten, custom fonts substituted with Google Fonts copies | ~80%; transitions/animations rarely round-trip, static layout + text + images survive |

**If you know the target is Google Workspace specifically:** for spreadsheets and presentations, the MS-Office equivalents (`.xlsx` / `.pptx`) actually round-trip with higher fidelity than their ODF counterparts. The default rule "ODF primary" still applies, but a user with a known Google-only target may explicitly request the MS-Office form. See `ms-office-authoring.md` §D.

## §C — Background recipe per ODF shape

The Void Navy `#000027` background MUST be applied via the format's native mechanism. Below is the minimum XML each format requires.

### §C.1 — `.odt` (text document)

Set the background on the **page layout** in `styles.xml`. Setting it only on the master page is not enough — Google Docs strips master-only backgrounds.

```xml
<office:automatic-styles>
  <style:page-layout style:name="Standard">
    <style:page-layout-properties
        fo:page-width="210mm"
        fo:page-height="297mm"
        style:print-orientation="portrait"
        fo:background-color="#000027"
        fo:margin-top="20mm"
        fo:margin-bottom="20mm"
        fo:margin-left="20mm"
        fo:margin-right="20mm">
      <style:background-image/>
    </style:page-layout-properties>
  </style:page-layout>
</office:automatic-styles>
```

Verify in LibreOffice via *Format → Page Style → Area → Color* — the swatch should read `#000027`.

### §C.2 — `.ods` (spreadsheet)

ODS has no per-page background by convention; the Void Navy mandate applies via **cell range fill** across every used cell. Apply `#000027` as the background fill to every cell in the active range (not just a sheet-level property).

```xml
<office:automatic-styles>
  <style:style style:name="ce-void-navy" style:family="table-cell">
    <style:table-cell-properties fo:background-color="#000027"/>
  </style:style>
</office:automatic-styles>
```

Reference `ce-void-navy` as the `table:style-name` on each `<table:table-cell>` (or apply via a default cell style for the sheet). For print, LibreOffice respects the cell fill; Excel honours the same on opening.

Optionally also set the sheet tab colour to a Spacecraft Software accent via `<table:table-properties tableooo:tab-color="#4B7EB0"/>` — purely cosmetic in LibreOffice's sheet-tab strip.

### §C.3 — `.odp` (presentation)

Set the background on the **slide master** so every slide inherits Void Navy:

```xml
<office:master-styles>
  <style:master-page style:name="Default" style:page-layout-name="Standard">
    <draw:page-thumbnail/>
    <!-- background applied via the draw-page-properties below -->
  </style:master-page>
</office:master-styles>

<office:automatic-styles>
  <style:style style:name="dp-master-bg" style:family="drawing-page">
    <style:drawing-page-properties
        draw:fill="solid"
        draw:fill-color="#000027"/>
  </style:style>
</office:automatic-styles>
```

Then verify on each individual slide — if a slide-layout declares its own `draw:fill`, it overrides the master. The safest approach: also set `draw:fill="solid"` + `draw:fill-color="#000027"` on every `<draw:page>` so the chain is unambiguous.

Slide geometry: 16:9 widescreen. In `styles.xml`:

```xml
<style:page-layout-properties
    fo:page-width="33.867cm"
    fo:page-height="19.05cm"
    style:print-orientation="landscape"/>
```

(33.867 × 19.05 cm ≈ 1920 × 1080 px at 144 dpi — standard LibreOffice Impress widescreen.)

## §D — Typography setup

Both Share Tech Mono and Inconsolata are OFL-licensed and may be embedded into ODF without licensing concern. Embedding is recommended for archival files; not strictly required for files that will only be viewed via Google Workspace (which re-renders from its own Google Fonts copies anyway).

Declare font faces in `content.xml`'s `<office:font-face-decls>`:

```xml
<office:font-face-decls>
  <style:font-face style:name="Share Tech Mono"
                   svg:font-family="'Share Tech Mono'"
                   style:font-pitch="fixed"
                   svg:font-face-uri="fonts/ShareTechMono-Regular.ttf"/>
  <style:font-face style:name="Inconsolata"
                   svg:font-family="'Inconsolata'"
                   style:font-pitch="fixed"
                   svg:font-face-uri="fonts/Inconsolata-Regular.ttf"/>
</office:font-face-decls>
```

Bundle the actual `.ttf` files inside the ODF container's `Fonts/` directory (an ODF file is a zip — `unzip -l report.odt` should show the font files alongside `content.xml` / `styles.xml` / `META-INF/manifest.xml`).

Map paragraph styles:

```xml
<style:style style:name="Heading_20_1" style:family="paragraph">
  <style:paragraph-properties fo:margin-top="0.6cm" fo:margin-bottom="0.2cm"/>
  <style:text-properties style:font-name="Share Tech Mono"
                         fo:font-size="16pt"
                         fo:font-weight="bold"
                         fo:color="#4B7EB0"/>
</style:style>

<style:style style:name="Heading_20_2" style:family="paragraph">
  <style:text-properties style:font-name="Share Tech Mono"
                         fo:font-size="14pt"
                         fo:font-weight="bold"
                         fo:color="#50FA7B"/>
</style:style>

<style:style style:name="Default_20_Paragraph_20_Style" style:family="paragraph">
  <style:text-properties style:font-name="Inconsolata"
                         fo:font-size="11pt"
                         fo:color="#D98E32"/>
</style:style>
```

For `.ods`, define analogous cell styles (`<style:style style:family="table-cell">`) referencing the same font names. For `.odp`, define presentation styles (`style:family="presentation"`) on the slide master.

## §E — Recommended toolchain per ODF shape

In rough order of preference:

### `.odt`

1. **LibreOffice headless** — convert from a markdown master or a templated ODF source:
   ```sh
   soffice --headless --convert-to odt --outdir build/ template.odt
   ```
2. **`odfpy`** (Python) — programmatic generation when content is data-driven. Stable API; produces ODF that LibreOffice opens cleanly.
3. **Hand-authored XML** — only when templating a small fixed structure; tedious to maintain.

### `.ods`

1. **`odfpy`** — best for raw structure and per-cell styling.
2. **LibreOffice headless** — for CSV → ODS conversion:
   ```sh
   soffice --headless --convert-to ods --outdir build/ data.csv
   ```
   Apply Spacecraft Software cell fills via a post-processing pass on the resulting `content.xml`.
3. **`pyexcel-ods3`** — simpler API than `odfpy` but limited styling support; OK for plain data dumps if the styling pass happens elsewhere.

### `.odp`

1. **LibreOffice headless from an ODP template** — strongly preferred. The slide-master XML is intricate enough that hand-building it from scratch is error-prone. Keep a `template.odp` in version control with the Spacecraft Software styles already configured; clone it and replace text via search-and-replace or `odfpy`.
2. **`odfpy`** — works for `.odp` but the API for slide layouts is thin; expect to write some raw XML.
3. **Avoid** PowerPoint's "Save As ODP" — produces ODF that LibreOffice opens but Google Slides sometimes mis-parses.

## §F — Pitfalls and gotchas

- **PowerPoint/Excel/Word "Save As ODF" exports** look like valid ODF but use idiosyncratic style names that Google Workspace's importer sometimes misinterprets. Always generate via LibreOffice or `odfpy`.
- **`META-INF/manifest.xml`** must list every embedded file (fonts, images). Missing manifest entries → silent stripping when opened in Google Workspace.
- **MIME type matters.** The `mimetype` file must be the first entry in the ODF zip, uncompressed (`zip -X0`). If you build the zip with default compression, some readers refuse to open it.
- **Tracked changes left in** — `<text:tracked-changes>` element in `content.xml`. Strip before shipping.
- **Embedded macros** are blocked by §3 of SKILL.md. Even if you embed them, sandboxing in modern readers will prompt the user; just don't.

## §G — Acceptance checklist (ODF deliverables)

In addition to SKILL.md §8 (general acceptance), specifically for ODF:

- [ ] `unzip -l <file>.odt` (or `.ods` / `.odp`) shows: `mimetype` first, `META-INF/manifest.xml`, `content.xml`, `styles.xml`, font files in `Fonts/`.
- [ ] `unzip -p <file>.odt content.xml | grep -c "Void Navy hex"` — well, `grep -c "#000027"` should return ≥ 1.
- [ ] Open in LibreOffice → Void Navy visible across the active surface (page / cells / slides).
- [ ] Upload to Google Drive → open in Docs/Sheets/Slides → Void Navy still visible. (Drag-and-drop, then "Open with" the appropriate Google app.)
- [ ] If MS Office is available on the host: open in Word/Excel/PowerPoint → Void Navy visible (acknowledged ~90/85/80% fidelity per §B).
- [ ] Headings use Share Tech Mono, body/cells use Inconsolata.
- [ ] GFM companion `.md` exists with the same base name (cross-reference `markdown-companion.md`).
